# Техническое задание: сайт-документохранилище Ассоциации Монтелиберо

## 1. Назначение проекта

Необходимо создать отдельный статический DOC-сайт для публикации документов из данного Git-репозитория.

Цель сайта:

- удобно показывать актуальные версии документов без интерфейсного шума GitHub;
- давать прямой доступ к исходным Markdown-файлам;
- давать доступ к историческим версиям документов по данным Git;
- сохранять привязку к структуре репозитория и истории изменений.

Репозиторий является первичным источником данных.

- Актуальное состояние документов определяется содержимым ветки `main`.
- Предыдущие состояния документов берутся из Git-истории.
- Документы хранятся как Markdown-файлы.
- `Meta.md` и `*.meta.md` являются служебными файлами с сопроводительной информацией.

Сайт должен быть ориентирован на спокойное чтение документов, навигацию по разделам и доступ к прошлым редакциям.

## 2. Общая модель данных

В репозитории есть два смысловых массива документов:

- `Internal/` — документы Ассоциации;
- `External/` — внешние и дружественные документы.

Для сайта принимается следующая модель:

- содержимое `Internal/` образует корень сайта;
- `External/` публикуется как виртуальный подкаталог в корне сайта: `/External/...`.

То есть на сайте не должно быть префикса `/Internal/`, но должен быть префикс `/External/`.

Примеры:

- `Internal/ProjectsAndPrograms/SkyNet.ru.md` -> `/ProjectsAndPrograms/SkyNet.ru.html`
- `Internal/Plans/2025/README.md` -> `/Plans/2025/`
- `External/MTLFund/TokensOffer.ru.md` -> `/External/MTLFund/TokensOffer.ru.html`

## 3. Какие файлы считаются публикуемыми сущностями

### 3.1. Основные документы

Публикуемыми документами считаются Markdown-файлы:

- `*.md`,
- кроме `README.md`,
- кроме `Meta.md`,
- кроме `*.meta.md`.

Именно для таких документов создаются:

- каноническая HTML-страница;
- канонический raw Markdown-файл;
- исторические слепки;
- страница истории документа.

### 3.2. README-файлы

`README.md` не считается самостоятельным документом первого класса.

Он используется только как содержимое index-страницы соответствующего каталога:

- если `README.md` есть, его содержимое рендерится в верхней части index-страницы;
- если `README.md` нет, index-страница всё равно создаётся автоматически.

### 3.3. Meta-файлы

`Meta.md` и `*.meta.md` не входят в общую навигацию как документы.

Для них создаются только HTML-представления:

- без raw-выгрузки;
- без исторических слепков;
- без отдельной страницы истории.

Meta-файлы должны быть доступны только по ссылкам со страниц канонических документов.

## 4. Целевое назначение сайта

Сайт должен решать следующие задачи:

- показать актуальную редакцию документа в чистом интерфейсе;
- дать скачать исходный Markdown без модификаций, байт в байт;
- дать перейти к историческим слепкам документа;
- показать историю правок документа в виде списка дат;
- дать перейти к коммиту на GitHub, в котором существует соответствующее состояние;
- сохранить структуру разделов и директорий репозитория;
- поддерживать документы на нескольких языках, если такие версии имеются в одном каталоге.

## 5. Каноническая адресация

### 5.1. Общий принцип

Канонический адрес документа должен соответствовать его пути в репозитории:

- относительно `Internal/`, если документ лежит в `Internal`;
- относительно корня сайта с префиксом `/External/`, если документ лежит в `External`.

Регистр символов в адресах сохраняется как в репозитории.

Никакой дополнительной нормализации путей не выполняется.

### 5.2. Канонические адреса документов

Для каждого основного документа должны существовать:

- HTML-страница документа: `.../FileName.ext.html`
- raw Markdown-файл: `.../FileName.ext.md`
- страница истории: `.../FileName.ext.history.html`

Примеры:

- `/Agreement/Agreement.ru.html`
- `/Agreement/Agreement.ru.md`
- `/Agreement/Agreement.ru.history.html`
- `/External/WorldVoluntaryist/VoluntaryistCovenant.en.html`

### 5.3. Канонические адреса каталогов

Для каждого каталога должна существовать index-страница по адресу каталога с завершающим `/`.

Примеры:

- `/`
- `/Plans/`
- `/Plans/2025/`
- `/External/`
- `/External/MTLFund/`

### 5.4. Адреса Meta-представлений

Meta-файлы публикуются в HTML-виде по отдельным адресам.

Точный формат имени должен отражать соответствующий исходный meta-файл и при этом явно показывать, что это meta-представление.

Правило выбора meta-файла для документа строгое и детерминированное:

- сначала ищется `FileName.meta.md` рядом с документом;
- если такого файла нет, ищется общий `Meta.md` в том же каталоге;
- если нет ни одного из них, ссылка `Meta` для документа не показывается.

Публичные адреса meta-представлений фиксируются так:

- `FileName.meta.md` -> `FileName.meta.html`;
- общий `Meta.md` каталога -> `Meta.html`.

Дополнительные правила:

- на канонической странице документа должна быть стабильная ссылка вида `Meta`;
- meta-страница должна открываться как HTML-представление;
- meta-страница не участвует в истории документа и не имеет raw-варианта;
- если один `Meta.md` относится к нескольким документам, допустимо, что разные документы ведут на одну и ту же страницу `Meta.html`.

## 6. Исторические слепки по SHA-256

### 6.1. Назначение

Для каждой версии каждого основного документа, обнаруженной в Git-истории этого файла, необходимо создать исторический слепок.

### 6.2. Адреса исторических слепков

Для каждого исторического состояния должны создаваться:

- `/[SHA-256].md`
- `/[SHA-256].html`

Где `SHA-256` — это SHA-256 от raw bytes соответствующего Markdown blob из Git.

### 6.3. Правило вычисления SHA-256

SHA-256 вычисляется строго по сырым байтам Git blob соответствующей версии Markdown-файла.

Это требование обязательно, чтобы пользователь мог:

- скачать `/[SHA-256].md`;
- самостоятельно посчитать SHA-256;
- получить точно такое же значение.

Никакая нормализация содержимого перед вычислением SHA-256 не допускается.

### 6.4. HTML-представление SHA-версии

`/[SHA-256].html` является не raw-файлом, а полноценной HTML-страницей сайта:

- с шаблоном;
- с навигацией;
- с метаданными;
- с информацией, что это исторический слепок;
- со ссылкой на актуальную каноническую версию документа.

При этом принимается осознанное проектное допущение:

- `SHA-256`-страница остаётся контентно-адресуемой страницей;
- в её служебном блоке допускается показывать одну конкретную дату фиксации и один конкретный commit;
- если один и тот же Markdown blob исторически встретится в нескольких файлах или в нескольких разных точках истории, сайт всё равно использует первый опубликованный вариант страницы;
- такие случаи считаются допустимой редкой аномалией первой версии и должны быть видны по warning в логах сборки.

### 6.5. Коллизии SHA-256

Если два разных исторических объекта приводят к одинаковому имени `SHA-256`, сборка не должна останавливаться.

Требуемое поведение:

- записать warning в лог сборки;
- сохранить первый сгенерированный объект;
- последующие совпадения не должны перезаписывать уже опубликованный файл;
- сборка должна завершаться успешно.

## 7. История документа

### 7.1. Источник истории

История документа строится только по коммитам, в которых менялся сам файл данного документа.

Коммиты, в которых менялись другие файлы, не должны попадать в историю документа.

### 7.2. Переименования

История должна тянуться через переименования и перемещения файла.

Для этого при сборке истории требуется учитывать Git follow-history.

### 7.3. Страница истории

Для каждого основного документа создаётся страница:

- `.../FileName.ext.history.html`

Страница истории должна содержать список редакций документа.

Для каждого пункта истории достаточно показать:

- дату и время фиксации версии;
- ссылку на соответствующую SHA-256 HTML-версию;
- ссылку на commit на GitHub.

Дополнительные подробности по коммиту на самой history-странице не обязательны.

### 7.4. Актуальная версия на history-странице

Актуальная версия документа также должна присутствовать в этой истории как одна из записей.

## 8. Связь канонической и исторических версий

### 8.1. Для канонической версии

На канонической странице документа должна быть ссылка `Permanent link`, ведущая на SHA-256-страницу актуального содержимого.

### 8.2. Для исторической SHA-версии

На исторической странице документа должна быть ссылка на канонический адрес документа, то есть на его актуальную редакцию.

Таким образом:

- канонический адрес всегда означает последнюю редакцию;
- SHA-адрес всегда означает конкретный исторический слепок.

## 9. Время и формат дат

Все даты и время в интерфейсе сайта должны показываться:

- в UTC;
- в формате `YYYY-MM-DD HH:mm`;
- без секунд.

Пример:

- `2026-04-21 13:47 UTC`

Это правило применяется:

- на страницах документов;
- на страницах истории;
- в индексах каталогов, если там выводится дата последнего изменения.

## 10. Индексы каталогов

### 10.1. Общий принцип

Index-страница должна существовать для каждого каталога в публикуемой структуре.

### 10.2. Если есть README.md

Если в каталоге есть `README.md`:

- его содержимое рендерится в верхней части index-страницы;
- ниже всё равно показывается обычный индекс непосредственного содержимого каталога.

### 10.3. Если README.md нет

Если в каталоге нет `README.md`, index-страница всё равно создаётся автоматически.

### 10.4. Что показывается в индексе

Index-страница показывает только непосредственных дочерних сущностей:

- подкаталоги;
- основные документы.

Рекурсивный вывод всего дерева вниз не требуется.

### 10.5. Что не показывается в индексе

В индексах каталогов не должны показываться:

- `Meta.md`;
- `*.meta.md`;
- непубликуемые не-Markdown файлы;
- служебные артефакты.

### 10.6. Что показывается для документа в индексе

Для документа в индексе достаточно показывать:

- ссылку на документ;
- дату последнего изменения документа.

Лишние служебные ссылки в индексе не нужны.

## 11. Ссылки внутри Markdown

### 11.1. Общий принцип

На сайте должны переписываться только внутренние ссылки, которые генератор способен однозначно интерпретировать.

Если ссылка не распознана надёжно, её нужно оставить как есть.

### 11.2. Правила переписывания

Если внутри Markdown встречаются внутренние ссылки, они должны преобразовываться так:

- ссылка на обычный документ `*.md` -> ссылка на его канонический `.html`;
- ссылка на `README.md` -> ссылка на адрес каталога;
- ссылка на `Meta.md` или `*.meta.md` -> ссылка на соответствующий `.meta.html`;
- внешние `http://` и `https://` ссылки не изменяются.

Для исторических SHA-страниц действует отдельное уточнение:

- внутренние ссылки внутри рендера исторического документа всегда ведут на канонические актуальные версии документов и разделов сайта;
- попытка восстанавливать ссылочные переходы между историческими версиями документов в первой версии проекта не требуется.

### 11.3. Якоря

Если у внутренней ссылки есть anchor-фрагмент, его нужно сохранять.

Пример:

- `SomeDoc.ru.md#section` -> `SomeDoc.ru.html#section`

Для корректной работы таких ссылок генератор должен использовать GitHub-style heading slugs, максимально совместимые с GitHub Flavored Markdown.

## 12. Языковые версии документа

### 12.1. Правило обнаружения

Языковые версии определяются только по имени файла в одном и том же каталоге.

Если в одном каталоге есть, например:

- `Agreement.ru.md`
- `Agreement.en.md`

то эти файлы считаются языковыми версиями одного документа.

Никакое более сложное сопоставление имён не требуется.

### 12.2. Отображение

На канонической странице документа нужно показывать ссылки на другие языковые версии, если они есть.

Формат может быть простым, например:

- `Other version: en`
- `Other version: ru`

Это требуется только для канонических страниц документов.

Для исторических SHA-страниц такие ссылки не нужны.

## 13. Что показывать на странице документа

### 13.1. Каноническая страница документа

Каноническая HTML-страница документа должна содержать:

- шапку сайта;
- подзаголовок раздела:
  - `Association Documents` для документов из `Internal`;
  - `External Documents` для документов из `External`;
- хлебные крошки;
- текущее каноническое имя файла как основной служебный заголовок страницы;
- дату фиксации текущей версии;
- ссылку на commit на GitHub;
- ссылку `Permanent link` на актуальную SHA-256 HTML-версию;
- ссылку на raw Markdown канонической версии;
- ссылку на history-страницу;
- ссылку `Meta`, если meta-файл для документа найден;
- ссылки на другие языковые версии, если они существуют;
- далее рендер самого Markdown-документа.

### 13.2. Историческая SHA-страница

Историческая HTML-страница документа должна содержать:

- ту же общую стилистику сайта;
- пометку, что это historical snapshot;
- каноническое имя файла документа;
- дату фиксации этой версии;
- ссылку на commit на GitHub;
- ссылку на канонический актуальный адрес документа;
- ссылку на raw `/[SHA-256].md`;
- далее рендер Markdown соответствующего исторического состояния.

## 14. Заголовки интерфейса и навигация

Интерфейс сайта должен быть на английском языке в простом и понятном международном варианте.

### 14.1. Главный заголовок сайта

Главный заголовок:

- `Montelibero Association Document Storage`

### 14.2. Подзаголовок сайта

Подзаголовок зависит от раздела:

- `Association Documents`
- `External Documents`

### 14.3. Хлебные крошки

Хлебные крошки должны присутствовать:

- на канонических страницах документов;
- на index-страницах каталогов;
- при необходимости на meta-страницах.

Они должны отражать путь документа или каталога в опубликованной структуре сайта.

## 15. Дизайн и визуальные требования

### 15.1. Общий стиль

Дизайн должен быть:

- минималистичным;
- спокойным;
- без лишних элементов интерфейса;
- удобным для чтения длинных текстов.

### 15.2. Тёмная тема

Сайт обязательно должен поддерживать тёмную тему.

### 15.3. Адаптивность

Сайт должен корректно работать как минимум:

- на десктопе;
- на мобильных устройствах.

### 15.4. Типографика

Нужна аккуратная типографика для Markdown-документов:

- удобочитаемая ширина текста;
- нормальные отступы;
- корректное отображение списков, цитат, заголовков, ссылок и кода;
- стилистически чистое отображение длинных юридических и организационных текстов.

## 16. Требования к печати

CSS должен содержать отдельные print-friendly правила.

При печати страницы документа:

- все лишние элементы интерфейса должны убираться;
- должен оставаться только короткий служебный заголовок печатной версии;
- в этом заголовке должны быть:
  - краткое название проекта;
  - название документа;
  - хлебные крошки / путь к документу;
- после заголовка должна идти горизонтальная линия;
- далее должен печататься только рендер документа;
- печатная версия должна выглядеть аккуратно и пригодно для чтения на бумаге или в PDF.

## 17. Markdown-рендеринг

Нужен Markdown-рендеринг, максимально близкий к GitHub Flavored Markdown.

Требование:

- ориентироваться на GitHub-подобный диалект Markdown;
- поддерживать обычные конструкции, реально используемые в документах репозитория;
- не вводить экзотических расширений без необходимости;
- использовать алгоритм генерации идентификаторов заголовков, совместимый с GitHub-style heading slugs.

## 18. Источник данных для commit-ссылок

Префикс URL GitHub-репозитория должен задаваться в настройках проекта явно, в конфигурации генератора.

Ссылки на commit должны строиться на основании этого настроенного префикса, а не выводиться динамически из локального `origin`.

## 19. Что не входит в первую версию

В первую версию не входят:

- публикация не-Markdown файлов;
- показ служебных meta-файлов в общей навигации;
- история meta-файлов;
- история README-файлов как отдельных документов;
- попытки интеллектуально сопоставлять документы с разными именами;
- рекурсивные деревья каталогов на index-страницах;
- редактирование документов через сайт.

## 20. Требования к публикации

Сайт должен быть пригоден для публикации через GitHub Pages.

Предпочтительная схема:

- сайт генерируется отдельно как статический набор файлов;
- GitHub Pages используется как хостинг результата сборки;
- логика построения истории и адресов реализуется в собственном генераторе.

## 21. Рекомендуемый технический подход к реализации

Рекомендуемый вариант реализации:

- собственный статический генератор;
- генерация на `Python`;
- получение истории и содержимого через `git`;
- шаблоны HTML через `Jinja2` или аналогичный шаблонизатор;
- Markdown-рендеринг через GFM-совместимую библиотеку;
- итоговая публикация в GitHub Pages через GitHub Actions.

Данный выбор объясняется тем, что проект является не обычным markdown-сайтом, а витриной Git-репозитория с:

- каноническими страницами;
- raw-выгрузкой;
- историческими SHA-снимками;
- history-страницами;
- специальной маршрутизацией для `Internal` и `External`.

### 21.1. Размещение кода сборщика

Вся техническая часть проекта, относящаяся к сборке сайта, должна быть по возможности сосредоточена в каталоге:

- `.viewer_builder/`

Это требование распространяется на:

- исходный код генератора;
- шаблоны;
- CSS и прочие ассеты сайта;
- конфигурацию проекта;
- вспомогательные локальные скрипты;
- прочие внутренние файлы, относящиеся к механике сборки.

Исключения допустимы только там, где инфраструктура GitHub или репозитория диктует фиксированное расположение файлов.

Примеры допустимых исключений:

- `.github/workflows/...`
- корневой `.gitignore`

### 21.2. Выходной каталог сборки

Результат сборки сайта не должен засорять рабочее дерево репозитория.

Допустим любой технически удобный вариант выходного каталога, если выполняются следующие условия:

- артефакты сборки не мешают работе с исходными документами;
- артефакты сборки не должны вручную сопровождаться как часть основного содержимого репозитория;
- итоговая публикация через GitHub Pages должна происходить автоматически из результата сборки.

Точное имя и расположение выходного каталога может быть выбрано на этапе реализации.

## 22. Критерии готовности первой версии

Первая версия считается готовой, если:

- сайт собирается из текущего репозитория автоматически;
- все основные документы из `Internal/` и `External/` доступны по каноническим HTML-адресам;
- raw Markdown-файлы доступны байт в байт по каноническим `.md`-адресам;
- для всех версий документов из истории доступны `/[SHA-256].md` и `/[SHA-256].html`;
- для каждого документа доступна history-страница;
- index-страницы существуют для всех каталогов;
- `README.md` рендерится в index-страницах, где он есть;
- meta-файлы доступны по ссылкам со страниц документов как HTML-представления;
- внутренние ссылки в Markdown корректно переписываются там, где они распознаются;
- тёмная тема работает;
- печатная версия документов оформлена отдельно и пригодна для использования;
- в логах сборки видны предупреждения о SHA-коллизиях, если такие возникнут;
- сайт можно опубликовать через GitHub Pages.
